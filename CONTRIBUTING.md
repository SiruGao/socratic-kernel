# Contributing to Socratic Kernel

感谢你愿意帮助构建一个增强判断力、而不是制造依赖的 AI 产品。

## 开始之前

Socratic Kernel 的功能判断需要同时满足三个标准：

1. **是否增强用户主体性**：用户是否仍然定义问题、选择标准并承担结论？
2. **是否产生必要价值**：功能是否解决真实问题，而不是单纯增加交互次数？
3. **是否尊重数据主权**：用户是否知道系统记录了什么，并能导出、修改或删除？

较大的功能、架构调整或认知模型变更，请先创建 Feature Request 讨论。小型修复、文案和无障碍改进可以直接提交 Pull Request。

## 本地开发

```bash
git clone https://github.com/SiruGao/socratic-kernel.git
cd socratic-kernel
python3 -m http.server 4173
```

打开 `http://localhost:4173`。

项目当前是零框架静态应用，不需要安装前端依赖。Node.js 仅用于测试和构建。

## 测试

提交前运行：

```bash
npm test
npm run build
```

同时手动检查：

- 首页、创建审议、逐题回答、完成审议和历史档案；
- JSON 导出、导入、单条删除和全部删除；
- 手机窄屏和桌面宽屏；
- 刷新页面后数据是否仍然存在；
- 离线缓存是否不会阻止新版本更新。

## 分支与提交

建议分支命名：

```text
feat/browser-extension
fix/import-validation
docs/inquiry-protocol
research/anti-sycophancy-benchmark
```

提交信息建议使用以下前缀：

- `feat:` 新功能
- `fix:` 缺陷修复
- `docs:` 文档
- `refactor:` 不改变行为的重构
- `test:` 测试
- `research:` 研究和评估材料
- `chore:` 工程维护

## Pull Request 要求

PR 应包含：

- 解决的问题；
- 对用户主体性的影响；
- 隐私和数据影响；
- 已完成的测试；
- 界面变化截图或录屏；
- 已知限制和后续工作。

请避免将无关改动放进同一个 PR。

## 苏格拉底式问题设计规范

新增问题不能只追求“深刻感”。一个合格的问题需要：

- 指向一个明确的推理对象；
- 用户可以用事实、理由或价值选择回答；
- 不预设用户存在心理问题；
- 不把某一种哲学框架包装成客观答案；
- 不因为用户表达情绪就立即进行逻辑审判；
- 能够说明为什么此时需要追问，而不是直接回答。

### 推荐分类

- 概念定义
- 事实与证据
- 可证伪性
- 隐藏前提
- 最强反方
- 价值冲突
- 利益相关者
- 长期后果
- 可逆实验
- 责任归属

## 禁止的产品模式

以下改动不会被接受：

- 用对话时长、连续使用或情绪依赖作为核心增长目标；
- 隐藏记录用户的精神、关系或价值判断数据；
- 根据认知画像投放广告或提高说服成功率；
- 将规则线索描述为心理诊断；
- 让 AI 在重要价值问题上直接替用户作出最终结论；
- 为了显得“敢于反驳”而机械地唱反调。

## 无障碍与语言

- 所有交互必须可以通过键盘完成；
- 保持足够颜色对比度；
- 不用颜色作为唯一状态信号；
- 中文文案优先清晰、具体，避免故意玄学化；
- 英文翻译应表达相同产品原则，而不是逐字翻译。

## 安全问题

涉及隐私泄露、数据删除失败、内容注入或部署安全的问题，请不要公开提供可利用细节。按照 [`SECURITY.md`](./SECURITY.md) 报告。

## 行为准则

参与项目即表示同意遵守 [`CODE_OF_CONDUCT.md`](./CODE_OF_CONDUCT.md)。
